Lead With These

Five talking points — the wins

What to put forward proudly. Each is real and demonstrable today.

  1. 1
    The core promise works end to end. 200+ photos → scored triage → a handful of hero candidates with a styled, client-ready report, in a single pass. No photo-by-photo manual review.
  2. 2
    We over-delivered on the brief. The SOW asked for skill files plus an agent config. We shipped that and a real automation layer: a wcm command-line tool, a folder watcher, macOS auto-start, and a branded HTML report. The skill still runs in a browser with zero setup.
  3. 3
    Smart two-stage cost design. Every image gets a cheap, fast triage; only the top ~12 reach the premium model. Speed and judgment without paying premium cost on every photo.
  4. 4
    Three delivery modes fit three real workflows. Ad-hoc from any device (Mica), local no-upload review on the Mac (Trent), and hands-free folder-drop automation for recurring shoots — the same engine behind all three.
  5. 5
    Calibratable to Trent's taste, not generic. A taste-reference template lets Trent teach the agent with real examples; output improves with each one rather than being frozen to a default rubric.
Be Honest About These

Five considerations — the risks

Surface these proactively. They build trust and frame the decisions below.

  1. 1
    Taste is still unproven at scale. The rubric is well-designed, but we have no validated accuracy numbers — we haven't measured how often the agent's hero matches Trent's pick on real shoots. This is the single biggest open question.
  2. 2
    Model versions are aging. The pipeline pins specific Opus/Haiku versions; newer, stronger models already exist. We need a deliberate upgrade-and-revalidate cadence rather than silent drift.
  3. 3
    Setup friction on the automated mode. API key, virtual environment, pip, launchd — it works, but it's brittle for a non-technical team. Recent work was already firefighting install issues. Someone has to own it when it breaks.
  4. 4
    Cost is real and currently unbounded. Every image hits the API. High-volume shoots and an always-on watcher mean recurring spend that nobody is metering or capping yet.
  5. 5
    Ownership and handoff are undefined. Per the SOW, IP transfers to WCM on payment and the contractor retains nothing. After handoff, who maintains the code, holds the API key, and runs model updates? No support arrangement exists yet.
What We Need From This Meeting

Five decisions to make

Each has a recommended default. The goal is to leave the room with these resolved.

  1. 1
    Which mode do we standardize on? Don't maintain all three forever. Pick the primary workflow — likely Cowork for Trent, the wcm pipeline for recurring shoots — and demote the rest to "also available."
  2. 2
    What is our accuracy bar, and how do we prove it? Agree a validation test: run the agent on past shoots where the hero is already known, measure the hit rate, and define "good enough to trust."
  3. 3
    Who pays for and holds the API key? WCM account vs. contractor pass-through, plus a monthly spend cap. A billing and security decision, not a technical one.
  4. 4
    Scope freeze, or fund Phase 2? Is this POC the deliverable (close out the SOW), or do we fund a calibration round, model upgrade, and a lower-setup hosted version? Scope changes require written agreement.
  5. 5
    Support model after handoff. Hand over the repo as-is, or a retainer covering maintenance, model updates, and break-fix. Decide before payment closes the engagement.

Westcoast Modern | SOW
v0.2 · Development of AI Skill Modules & Agent Framework

Client: Westcoast Modern  ·  Contractor: Matthew A. Cottrell

1. Project Objective

To design, document, and implement modular AI skill sets (skills.md) and a cohesive agent configuration (agent.md) to automate and standardize the selection of the hero image from a batch of real estate property photos.

2. Deliverables

  • Skill Modules (skills.md): Development of distinct AI skill files. Each file includes:
    • Logic Definition — clear, repeatable logic paths for specific tasks.
    • Input/Output Schemas — standardized formats for processing data.
    • Contextual Constraints — rules for maintaining brand voice, technical accuracy, and design guidelines.
  • Agent Configuration (agent.md): Development of a primary agent framework containing:
    • Orchestration Logic — instructions on how the agent calls upon the specific skills.md files.
    • Persona / Role Definition — behavioral parameters for the agent.
    • Execution Workflow — the end-to-end process from input trigger to final output.

3. Timeline

  • Phase 1 (Drafting) — initial structure of agent.md.
  • Phase 2 (Skills Development) — delivery of draft skills.md files.
  • Phase 3 (Review / Refinement) — iterative adjustments based on client feedback.

Consulting Agreement

  • Services — contractor performs the services outlined in the SOW. Any additions or modifications to scope must be agreed in writing.
  • Fees & Billing — flat fee TBD; payment Net 15 days upon receipt of invoice; invoices via email; pre-approved out-of-pocket expenses reimbursed.
  • Intellectual Property — upon full payment, all work produced (including skills.md and agent.md content) becomes the property of the Client; the contractor retains no rights to the specific implementation details.
  • Confidentiality — contractor holds all proprietary business data, code structures, and project-related AI logs in strict confidence.
  • Termination — either party may terminate with 10 days' written notice; on termination, the Client pays half the agreed rate.
